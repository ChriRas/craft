# Plugin Architecture — Universal Coding Workflow

> Concrete architecture spec derived from the brainstorm session (see [`brainstorm-decisions.md`](./brainstorm-decisions.md)). Use this as the build blueprint.

---

## 1. Mission

A Claude Code plugin that wraps the user's universal coding workflow — the 8-phase Thorstensen loop, hardened with explicit knowledge management, autonomy controls, and bug-verification discipline — so it works identically across shell scripts, libraries, REST APIs, full-stack apps, data pipelines, and infrastructure code.

**Core principle:** *Universality with constant control.* Same workflow, same control surface, regardless of stack.

## 2. Two-Tier Model

| Tier | Lives at | Owns | Updates via |
|---|---|---|---|
| **Plugin (universal shell)** | `~/.claude/plugins/<plugin-name>/` | Workflow logic, universal commands, phase navigation, autonomy matrix, plugin-shipped skills | Plugin update mechanism |
| **Project-local (specialist body)** | `<repo>/.claude/` | Language-/stack-specific skills, agents, project tooling commands, allowlists, project knowledge files | Editing the project repo |

**Precedence:** project-local wins on name conflict (Claude Code default). After `/onboard` migration, overlapping legacy commands are moved to `.claude/commands/_legacy/`, so plugin commands become active.

**Universal vs. specialist:** Plugin commands lazy-load project-local skills/agents at runtime (e.g., `/execute` on a PHP file loads project's `developer` skill without redefining it).

**Notes on conventions (verified against the official plugin spec):**
- `plugin.json` must live in `.claude-plugin/`, not at the plugin root.
- `commands/` is technically the legacy slash-command directory; `skills/` is preferred for new functionality. We keep `commands/` because our 17 entry points are user-invoked verbs (`/prime`, `/plan`, `/commit`), which is the command pattern. Skills hold capability/knowledge content.
- Hooks live in a single `hooks/hooks.json` registry per plugin, not as individual scripts (though `hooks.json` may reference scripts via `${CLAUDE_PLUGIN_ROOT}/hooks/<name>.sh`).
- `templates/` is not a reserved directory in the Claude Code plugin spec; the `onboard` skill references files inside it explicitly.

## 5. Commands

All commands are language- and stack-agnostic. Project-specifics come from loaded project-local skills.

| Command | Phase / Purpose | Default Autonomy | Key behaviors | Refs |
|---|---|---|---|---|
| `/onboard` | First-time setup or migration | 0 | Auto-detects existing `.claude/` content; offers heuristic-first or grill-me intensive mode; per-class confirmation with override; generates `intent.md`/`rules.md`/`roadmap.md`; moves conflicting legacy commands to `_legacy/`. | D16, D18, D20 |
| `/prime` | Session start | 0 | Strict tool health check (abort on missing); auto-activates context-mode + currency check; loads `intent.md`+`rules.md`; runs Rules↔State drift check; scans `.claude/plans/`; outputs **Recommended Next Action**. | D17 |
| `/plan <feature>` | Phase 3 — planning | 1 | Dialogically forces the three universal questions: trigger, observable effect, end-to-end test strategy. Writes `slice-NNN-<slug>.md` with `plugin-version` frontmatter. | D7, D10 |
| `/continue` | Resume active slice | inherits phase | Reads slice file → resumes at recorded phase/sub-task. | D7 |
| `/execute` | Phase 4 — implementation | 2 | Implements sub-tasks; lazy-loads project specialist skills; runs tests silently (Level 3); bundles at sub-task boundary or 30k-token brake. | D13, D14, D15 |
| `/test` | Phase 5 — user testing | 1 / 0 | 5a: agent prepares demo-setup from slice trigger. 5b: user exercises. 5c: structured feedback `[W]/[B]/[U]` — `B` auto-triggers `/debug`, `U` asks "what should differ" directly. | D21 |
| `/recap` | Phase 6 — recap | 1 | Plain-text What/Why/Decisions; offers Mermaid diagram for complex slices (>3 modules). | D22 |
| `/refactor` | Phase 7 — refactoring | 1 | Asks the three Thorstensen prompts dialogically; max 2–3 items per slice; bigger refactor = new slice. | D22 |
| `/commit` | Phase 8 — commit & cleanup | 1 | Atomic commit split; Conventional Commits + `Slice:` footer; walks Decisions list with `[K]/[I]/[R]/[D]` promotion dialog; deletes active plan file; writes archive entry. | D9, D22, D8 |
| `/debug <description>` | Bug self-verification | 0→2→0 | 4-step loop: ALIGN (Level 0) → PROTOCOL (Level 0) → AUTONOMOUS LOOP (Level 2, 15k token brake, max 5 attempts) → ESCALATION (Level 0). Auto-offered at ≥2 fix attempts in slice. | D19 |
| `/handoff` | Fresh-context restart | 1 | Summarizes problem + attempts into slice plan's `## Handoff` section; user starts fresh chat; new session's `/prime` reloads. | D19 |
| `/brainstorm <topic>` | Phase 1 | 0 | Wraps `skills/brainstorm/`; dialogic, one question at a time. | adopted |
| `/grill-me <plan>` | Phase 2 | 0 | Wraps `skills/grill-me/`; interview-style alignment. | adopted |
| `/pause` | Mid-slice pause | n/a | Saves state, slice remains open. | D22 |
| `/abort <slice>` | Abandon slice | 0 | Confirms, deletes plan file by default. | D22 |
| `/intent-update` | Manual intent edit | 0 | Agent proposes diff, human confirms. | D5 |
| `/status` | Lightweight slice overview | 3 | Reads `plans/` and reports — same as `/prime`'s recap section, without re-loading everything. | — |

## 6. Skills (plugin-shipped, universal)

| Skill | Purpose | Used by |
|---|---|---|
| `workflow` | The 8-phase loop methodology itself. Loaded by every phase command to ensure phase-correctness. | All phase commands |
| `self-verify` | Bug verification protocol. Defines verification format and the autonomous fix loop. | `/debug` |
| `brainstorm` | Structured ideation. Adopted from `skills-main`. | `/brainstorm`, `/onboard` (grill-me-intensive mode handoff) |
| `grill-me` | Interview-style stress-testing. Adopted from `skills-main`. | `/grill-me`, `/onboard` (intensive mode), `/plan` (Phase 3 dialogic questions) |
| `agent-browser` | Browser automation wrapper (commands: `agent-browser open/back/screenshot/...`). | `/test` Phase 5a for web-stack demos |

## 7. Knowledge Model (recap)

| Layer | Content | Source of truth | Persisted in |
|---|---|---|---|
| **State** | What exists | Code, git, configs | Not duplicated; read on demand |
| **Intent** | What we want & why | Human-authored | `.claude/project/intent.md` |
| **Rules** | Operational distillation | Derived from State + Intent, **materialized** | `.claude/project/rules.md` |

Discipline: `/prime` validates Rules ↔ State on every run; drift is reported, never silently corrected.

## 8. Autonomy Matrix (recap)

#### Phase defaults
Phase 1 = 0, Phase 2 = 0, Phase 3 = 1, Phase 4 = 2, Phase 5 = 1, Phase 6 = 1, Phase 7 = 1, Phase 8 = 1.

#### Action-type force-overrides
- Reads / lint / status / file-grep = **Level 3** (silent, summarized in bundle)
- Code edit inside plan scope = phase default (~2)
- Code edit outside plan scope = **Level 1** (plan is contract)
- `rules.md` / `intent.md` mutation = **Level 0** (sacred)
- Git commit = **Level 1**
- Push / PR / deploy = **Level 0** (external, irreversible)
- Branch delete / force-push = **Level 0**

#### Bundling rules
- Bundle boundary: sub-task end OR 30k-token brake (15k for `/debug`).
- Auto-continue with abort option at Level 2.
- Level-3 actions summarized in bundles (one line: "read 12 files, ran 3 lint checks").

## 9. Rule Conflict Resolution

When `/prime` reports drift or human/agent disagreement during the loop:

| Conflict | Default winner | Action |
|---|---|---|
| State vs. Intent | Intent | Agent builds toward Intent (loop purpose) |
| Rules vs. State | Rules | Agent corrects State autonomously |
| Rules vs. Intent | Rules (default); **override** offered to human | Agent flags, asks |

**Three-stage rule override:**
- **Bend** — single exchange, no persistence.
- **Override** — slice-scoped, recorded in slice plan, cleared at Phase 8.
- **Repeal** — permanent, edits `rules.md`.

## 10. End-to-End Walkthrough (one slice)

```
[Session start]
  └─ SessionStart hook fires
     └─ /prime (auto)
        ├─ tool health: context-mode ✓ (activated), agent-browser ✓, git ✓, gh ✓
        ├─ loads intent.md + rules.md
        ├─ Rules ↔ State drift: clean
        ├─ scans .claude/plans/ → no active slice
        └─ recommends: /plan to start

[User: /plan "reservation button"]
  └─ Phase 3 (Planning, Level 1)
     ├─ dialogic: trigger? observable effect? test strategy?
     ├─ generates slice-007-reservation-button.md (status: planning)
     └─ frontmatter: plugin-version: 1.4.2

[User: /execute]
  └─ Phase 4 (Implementation, Level 2)
     ├─ loads project's `developer` skill (PHP/Laravel)
     ├─ sub-task 1/5: backend endpoint
     │    └─ bundles: ✓ added POST /reservations, tests green
     ├─ sub-task 2/5: frontend button component
     │    └─ ⚠ 2nd fix attempt on same symptom → offers /debug
     │       └─ /debug enters protocol, autonomous loop, fixes in 2 attempts
     ├─ sub-task 3–5 continue with auto-notify bundles
     └─ Phase-end bundle: all sub-tasks done, tests green

[/test invoked automatically]
  └─ Phase 5
     ├─ 5a: "Open http://localhost:8380/reservations, click button, expect toast"
     ├─ 5b: user clicks, finds button color wrong
     ├─ 5c: user → [U] "color should be neon-pink"
     │    └─ agent asks directly: "Which exact shade? Hex?"
     └─ iterate 5a–5c until [W]

[/recap]
  └─ Phase 6 (Level 1)
     ├─ plain-text recap (not complex enough for Mermaid)
     └─ user confirms

[/refactor]
  └─ Phase 7 (Level 1)
     ├─ three Thorstensen prompts
     ├─ user picks 2 refactor items
     └─ agent applies, tests stay green

[/commit]
  └─ Phase 8 (Level 1)
     ├─ atomic commit split, conventional commits + Slice: slice-007
     ├─ Decisions promotion dialog:
     │    "Optimistic UI Update" → [I]ntent  → appended to intent.md (human confirms)
     │    "neon-pink for action buttons" → [R]ules → added to rules.md/conventions
     │    "no WebSocket for MVP" → [K]eep in archive
     ├─ writes .claude/project/slices/slice-007-reservation-button.md
     └─ deletes .claude/plans/slice-007-reservation-button.md

[Session end → next /prime]
  └─ no active slices → recommend /plan
```

## 11. Tool Dependencies

| Tool | Plugin requires | If missing |
|---|---|---|
| context-mode | Yes | `/prime` aborts with install instructions |
| agent-browser | Yes | `/prime` aborts with install instructions |
| git | Yes | `/prime` aborts with install instructions |
| gh | Yes | `/prime` aborts with install instructions |

**All missing tools produce the same UX**: clear error + concrete install command for the user's OS. Rationale (D17): strict over graceful — token efficiency and reliability matter more than soft failures, but a hard abort without guidance is hostile.

## 12. Hooks

Plugin ships a centralized hook registry at `hooks/hooks.json`. Plugin hooks are auto-registered on install — the user does not edit their own `settings.json`.

```json
{
  "description": "Auto-prime on session start when the project is onboarded.",
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "bash ${CLAUDE_PLUGIN_ROOT}/hooks/session-start.sh",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

The `hooks/session-start.sh` script:
- If `.claude/project/intent.md` exists in the current project → emits the `/prime` invocation.
- Otherwise → emits a one-line nudge: "Project not onboarded — run `/onboard`".

`${CLAUDE_PLUGIN_ROOT}` resolves to the installed plugin directory at runtime.

## 13. Open Implementation Questions

Items deliberately deferred from brainstorm — resolve during build:

1. **Plugin distribution** — Resolved: Git repo (this directory) for development; later submit to `claude-plugins-official` marketplace via https://clau.de/plugin-directory-submission.
2. **Templates language** — English (matches the user's explicit preference: communication German, artifacts English).
3. **Slice numbering across sessions** — Continuous `slice-NNN`. Need atomic allocation: probably a small counter file at `.claude/plans/.next-id` updated by `/plan`.
4. **Drift check implementation** — Bash scripts in `skills/workflow/scripts/` invoked from `/prime`; semantic checks added later as skill content matures.
5. **Skill activation precedence** — Project-local should win over plugin-shipped (mirrors command precedence). Verify against the spec during build.
6. **Bundle-output rendering** — exact format in Claude Code terminal (color, spacing); needs prototype.
7. **`agent-browser` skill source** — pull from existing `real_live_projekt` version, or upstream source? Versioning?
8. **Tool-dependency self-check at install** — the Claude Code plugin spec does **not** support declarative dependency-on-tools (e.g., `requires: context-mode, agent-browser, gh`). Therefore, the missing-tool detection has to happen at `/prime` runtime (D17 already specifies this — confirmed correct approach).
9. **Plugin version stamping** — the plugin cannot detect its own version at runtime via the spec. For the `plugin-version` field in slice plan frontmatter (D22-S6), `/plan` reads `.claude-plugin/plugin.json` directly via the workflow skill.
10. **Capability / personality autoload (parked for future brainstorm)** — where does the library of language/framework personalities live, and how is it lazy-loaded?
   - Distinct from project-local specialist skills: a personality like "Symfony Senior PHP Developer" or "Django Backend Engineer" is **portable across projects** that share the same framework, so duplicating it in every `.claude/skills/` is wrong.
   - Distinct from the plugin's universal shell: personalities are language- and framework-specific.
   - Candidate locations:
     - Plugin ships a `personalities/` library (versioned, updatable with the plugin).
     - User-global `~/.claude/personalities/` (shared across all the user's projects, separate from plugin updates).
     - Hybrid: plugin ships a baseline, user-global extends/overrides.
   - Distinction within a stack: e.g., "PHP general" vs. "PHP Symfony" vs. "PHP Laravel" — personalities likely need composition or inheritance ("Symfony" extends "PHP general").
   - Trigger: agent loads the right personality on demand based on `intent.md` / detected stack / explicit `/persona <name>` invocation.
   - **Decision deferred** — needs its own brainstorm thread before implementation. Likely introduces a third tier between the plugin and project-local layers.
